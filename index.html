Coupon Card & Point-of-Sale Redemption

A physical sales channel for Chinmaya Clix. Pre-generated, non-guessable cards unlock a mapped product (a TVOD series or an SVOD plan). Cards are tracked end-to-end — generated → printed → assigned → redeemed — so stock and revenue reconcile exactly. This page is the developer KT: every screen, every case, and the full diagram set.

🆕 New CMS "POS" module 🔐 New POS role 🎟️ Batch + per-code lifecycle 📧 Email + invoice on assign 🍿 Auto-grant + portal popup 📊 Reconciliation in scope ₹ INR only · v1
§1

Overview

Today Clix is online-only. Many devotees buy in person (Chinmaya Vani store, CCMT, events). This feature lets staff sell Clix access over the counter: a card carries a unique code, an admin binds it to the customer's email and the system raises an invoice, and the customer activates by simply signing in — access unlocks automatically.

🎯 What we are building (v1)

  • A new POS module in the CMS sidebar.
  • Batch generation of N unique codes mapped to one series or plan.
  • Per-code status list with mark-as-printed (multi-select).
  • Assign a printed code to a customer email → invoice (INR) + email with PDF.
  • Auto-grant access; portal popup notifies the customer (gifting pattern).
  • Orders visibility + a reconciliation report with export.
  • A dedicated POS role limited to the POS page.

🧱 Reuse — we are not starting from scratch

  • Gifting feature is the reference pattern: email-bound recipients, AWAITING_CLAIM → REDEEMED, idempotent row-locked fulfillment, invoice + receipt, the GiftPopup.
  • Entitlement: same create_purchase / gift-subscription paths that grant access today.
  • Orders page already lists gifts via a dedicated source — we add a "POS Card" source the same way.
  • Emails: clone send_gift_received_* (Hari Om house style).
  • Toasts / errors: sonner + getApiErrorMessage; block-rules like check_block_rules.
§2

Code lifecycle (states)

Simplified from the spec's 6 states. No Allocated (single store), no Voided / Refunded / Expired (out of scope for v1). The one nuance: assigning to a new email parks the code at Awaiting claim until the customer registers; assigning to an existing user grants access immediately (Redeemed).

Generated → Printed → Assigned · Awaiting claim → Redeemed · Active (existing user skips straight to Redeemed)
Generated code created in a batch, not yet printed Printed card printed & verified — now assignable Awaiting claim bound to a not-yet-registered email Redeemed access granted (terminal · irreversible)

🔁 Why the split matters

Reconciliation (§7) explicitly watches Printed − Assigned (unsold stock) and Assigned − Redeemed (paid but not yet activated). Collapsing the two assign states would hide the "paid but not activated" gap. FR-7.3§8

📋 Requirements & cases

  • One batch maps to exactly one product — a series or a plan. FR-1.2D1
  • Generate N unique, non-guessable codes in a single action; codes never reissued. FR-1.1FR-1.4
  • Code format = RANDOMPREFIX-RANDOMSUFFIX; prefix shared per batch, suffix unique per card. Collision-checked for global uniqueness. FR-6.2

  • No expiry field in v1 (deliberately dropped). D3 ✗

Screen 5

Blocking & toasts

Every action confirms with a toast on success and explains on failure. The key block: a customer who already has access to the content can't be assigned again — the backend returns a clear message that the toast surfaces.

Click any button — toasts appear bottom-right, exactly as sonner renders them in the app.

🧩 Backend messages (messages.py)

  • CARD_RECIPIENT_ALREADY_HAS_ACCESS — "This user already has access to this content."
  • CARD_ALREADY_ASSIGNED — "This code has already been assigned."
  • CARD_NOT_PRINTED — "…must be marked printed before it can be assigned."
  • CARD_INVALID_CODE — "Invalid or unknown code."

🔎 How "already has access" is detected

  • TVOD: existing completed Purchase for the series, or active plan that includes it.
  • SVOD: active/trialing Subscription.
  • Mirrors gift check_block_rules; raised as BadRequestError → {status:"error",code:400,message}.

Data model

Two new tables — card_batch and card — plus links to the existing invoice, purchase and subscription rows the card fulfils. (Full ER diagram is in View Flowchart & Diagrams.)

🗃️ card_batch

  • id, organization_id
  • product_type tvod | svod
  • content_id (series) / plan_id
  • prefix — random, shared per batch
  • quantity, currency='INR'
  • created_at, created_by

🎟️ card

  • id, batch_id
  • code — PREFIX-SUFFIX, globally unique
  • status generated | printed | awaiting_claim | redeemed
  • assigned_email, assigned_user_id, recipient_name, recipient_mobile
  • invoice_id, fulfilled_purchase_id, fulfilled_subscription_id
  • audit: printed_at/by, assigned_at/by, redeemed_at, seen_by_recipient_at
§5

Edge cases & business rules

✔ Handled

  • Duplicate-assign on a sold code → blocked + clear toast. FR-4.5
  • Customer already owns the product → blocked. FR-6.3
  • Concurrent assign of one code → row lock, no double-sell. NFR
  • Once assigned, the binding is irreversible — no edit / re-assign.
  • Globally-unique, non-guessable codes (collision-checked). FR-1.4FR-6.2
  • Product mismatch impossible — a code unlocks only its mapped product. FR-6.3

⚠ Notes / trade-offs

  • Codes stored plain (not hashed) — they must be displayed/printed. NFR
  • "Staff" in reports = the Owner/Admin who assigned (assign is admin-only, not counter POS).
  • Email mismatch handled by binding — only the bound email's account receives access.
  • Print step = checkbox + simple confirm (not a multi-item checklist).

Assign sequence

Row-locked, idempotent — mirrors gift fulfillment. Blocks if the user already owns the product.

sequenceDiagram
  autonumber
  actor A as Admin
  participant UI as CMS POS Page
  participant API as Backend API
  participant DB as Database
  participant M as Email (SES)
  actor C as Customer
  A->>UI: Enter code + email, click Assign
  UI->>API: POST /pos/cards/{id}/assign {email}
  API->>DB: SELECT card ... FOR UPDATE (lock)
  API->>DB: lookup user by email
  alt user exists AND already owns content
    API-->>UI: 400 "User already has this access"
    UI-->>A: error toast
  else OK
    alt existing user
      API->>DB: create Purchase / Subscription (grant)
      API->>DB: create INR invoice (live price)
      API->>M: access email + invoice PDF
      API->>DB: status = Redeemed
    else new user
      API->>DB: status = Awaiting claim
      API->>M: sign-up + access email
    end
    API-->>UI: 200 success
    UI-->>A: success toast
  end
  C->>API: register / login (if new)
  API->>DB: claim binding -> grant access
  C->>API: next portal visit
  API-->>C: popup "You now have access"
